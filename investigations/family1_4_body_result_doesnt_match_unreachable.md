# Why `BodyResultDoesntMatch` is unreachable for the int/bool lambda mismatch

> **Status:** Investigation complete. No code fix needed; this is by-design behavior.

## Test under investigation

`AfterRegionsErrorTests.scala "Lambda body type mismatches anonymous interface return type"`

```vale
interface AFunction1<P Ref> {
  func __call(virtual this &AFunction1<P>, a P) int;
}
exported func main() {
  arr = AFunction1<int>((_) => { true });   // bool body, expected int
}
```

The test asserts `Err(BodyResultDoesntMatch(...))`. It actually produces a `Couldn't find a suitable function AFunction1(...)` overload-resolution error whose inner reason is `ReturnTypeConflictInConclusionResolve(expected=i32, actualReturn=bool)`.

## Why the body-vs-return comparison is unreachable here

Two reasons stack:

1. **Lambda body has no explicit expected return type.** Per LAGTNGZ ("Lambdas Are
   Generic Templates Not Generics"), the lambda's `__call` body is compiled with
   `maybeExplicitReturnCoord = None`. `evaluateFunctionBody`'s mismatch path only
   runs in the `Some(expectedResultType)` branch (FunctionBodyCompiler.scala:194-204).
   So the lambda alone can never trigger `BodyResultDoesntMatch`.

2. **The forwarder, which *does* have an explicit return type, is gated behind
   successful constructor inference.** The forwarder is generated by
   `AnonymousInterfaceMacro#makeForwarderFunction` and inherits the abstract
   method's return rune (`Some(inheritedReturnRune)` at line 542). If it were
   compiled with a body whose result coord didn't convert to that rune's value,
   we'd hit `BodyResultDoesntMatch`. But the forwarder is only compiled *after*
   the substruct constructor's inference succeeds, and inference fails first at
   the substruct's `__call` bound (the `ResolveSR` rule emitted by the macro
   at lines 332-334 of AnonymousInterfaceMacro.scala). That bound check uses
   `delegate.resolveFunction` to find the lambda's `__call` and compares its
   actual return type to the expected one — which is exactly the
   `ReturnTypeConflictInConclusionResolve` we observe.

In short: the substruct's bound system catches the mismatch *strictly earlier*
than any body-vs-return-type comparison could. By design.

## Could we engineer a case where the forwarder body fails?

Probably not via lambdas. The forwarder's body is mechanically generated as
`self.0(args)` — it just calls the lambda. If the lambda's return type
disagrees with the forwarder's declared return type, inference rejects the
substruct first; the body never compiles. To reach `BodyResultDoesntMatch` for
a forwarder would require the forwarder's body to be *correctly inferable* but
*incorrectly typed*, which contradicts how it's mechanically built.

`BodyResultDoesntMatch` for plain (non-anonymous-substruct) functions remains
the realistic path — already covered by `CompilerTests.scala:497 "Reports
mismatched return type when expecting void"`.

## Recommendation

Either:

- **Delete the mismatch test.** The int/bool detection is already covered by
  the overload-resolution path (and the substruct bound check is well-tested by
  the broader integration suite).
- **Change the assertion** to expect the realistic error
  (`CouldntFindFunctionToCallT` containing
  `ReturnTypeConflictInConclusionResolve` with the right return coords). This
  would lock in the substruct-bound-check behavior as the contract.

Either is reasonable. The original handoff's framing — "ensure the body-check
fires for synthesized forwarders" — turns out not to match how the compiler is
designed; the bound check supersedes the body check in this code path.
