# Cluster #2 (Bucket B): OverloadResolver bound-prefix mismatch

(Id-shorthand notation per `docs/background/id-shorthand-notation.md`.)

**Status**: cluster #2's main case **fixed**. One-line `StructCompilerCore.scala` change in working tree. Backend suite: 175 → 180/198. Zero regressions. Bucket B residual (foreach/stdlib_hashmap/stdlib_hashset), Bucket C, and Bucket D are unrelated sub-clusters and remain.

## The failure

`OverloadResolver.scala`'s `narrowDownCallableOverloads`:

```scala
sortedByNameLength.tail.foreach(otherCandidate => {
  vassert(otherCandidate._2.id.initSteps.startsWith(shortestCandidate.id.initSteps))
})
```

Fires when overload resolution for `__call` returns multiple `FunctionBoundNameT` candidates whose `initSteps` don't share a structural prefix. Originally observed on `upcastif`, `interfacemutreturnexport`, `foreach`, `stdlib_hashmap`, `stdlib_hashset`.

## Minimum repro

```vale
interface I {
  func moo(virtual this &I) int;
}
exported func main() int { return 0; }
```

Empty `main`, no `impl`, no concrete struct. The crash is entirely in compiler-synthesized code that runs whether or not the program touches it. `AnonymousInterfaceMacro` fires for every non-sealed interface and auto-generates a sibling anonymous substruct (`anon:I`) — the "boxed callable" wrapper — plus its constructor, drop, and forwarder methods. The substruct's auto-generated `drop` calls `__call` on its borrowed member; resolving that `__call` is where the assertion fires.

(Adding `sealed` to `interface I` makes `AnonymousInterfaceMacro` early-out and skips the whole machinery. Every cluster #2/#3 test happens to declare a non-sealed interface.)

## Notation primer

- **`anon:I`** — the anonymous substruct template of `I` (`AnonymousSubstructTemplateNameT(InterfaceTemplateNameT(I))`). One name token; `I` is a payload-style qualifier.
- **`anon:I<^anon:I$functor:moo>`** — the same substruct, instantiated with its single member-rune placeholder (the `moo`-functor).
- **`drop:0`** — `AnonymousInterfaceMacro` synthesizes drop with `FileCoordinate(<file>, 0)` since there's no source line to point at. Bare `drop:0` is the template name; `drop:0<...>(...)` is the instantiated form.

## Root cause

drop:0's body invokes `__call` on its borrowed member; the resolver finds two candidates:

| | id | declared on |
|---|---|---|
| **shortest** | `anon:I.bound:__call<>(&anon:I$functor:moo)` | substruct's own `where exists __call(...)` contract (auto-generated by `AnonymousInterfaceMacro`) |
| **longer** | `anon:I<^anon:I$functor:moo>.drop:0.bound:__call<>(&anon:I$functor:moo)` | drop:0's denizen, via NBIFP from drop:0's `^anon:I<^anon:I$functor:moo>` parameter |

`startsWith` fails at step 0: shortest's only step is `anon:I` (template form), longer's first step is `anon:I<^anon:I$functor:moo>` (instantiated form). Same denizen at the template level; different Scala objects.

This branch's @SMLRZ/Fix-#2 (commit `76cd7e0c`) is what made the parent step take instantiated form; on master it was `anon:I` everywhere and the assertion was a structural no-op.

### Why two `__call` declarations exist (NBIFP, by design)

Per `docs/Generics.md`'s "Need Bound Information From Parameters (NBIFP)": when a function takes a parameter whose type carries `where exists __call(...)`, the typing pass automatically imports `__call` into the function's own denizen as a prototype. This is what makes monomorphization self-contained — when the monomorphizer stamps drop:0 for a concrete callable, drop:0's bound-args (including the local NBIFP-imported `__call`) are what it consults.

So the two declarations exist for sound reasons:
- The **substruct** declares `__call` on its own env so callers of the substruct can find it via param-type-env lookup.
- **drop:0** has its own copy on its own env so its monomorphization works without needing to walk up.

### Why both surface during overload resolution

`OverloadResolver.getCandidateBanners` calls `lookupAllWithImpreciseName` from drop:0's call-site env. That triggers `EnvironmentHelper.lookupWithImpreciseNameInner`'s parent recursion (`result ++ parent.lookupWithImpreciseNameInner(...)`).

drop:0's parent chain at the time of the bug: **drop:0 → structInnerEnv (anon:I instantiated) → outerEnv (anon:I template) → packageEnv**. Walking up *traverses the substruct's CitizenEnvironmentT*, which contains the substruct's contract bound. drop:0's local NBIFP-imported bound surfaces at step 1 of the walk; the substruct's contract bound surfaces a few steps later. Both accumulate in the result; the resolver tries to narrow them; `startsWith` asserts they share a structural prefix; they don't.

## The applied fix (site-3 architectural)

The duplicate exists at three places, each with its own role:

- **Site 1** (substruct stores `__call` on its env): required for callers' param-env lookup. Don't touch.
- **Site 2** (NBIFP imports `__call` into drop:0's env): required for drop:0's monomorphization. Don't touch.
- **Site 3** (drop:0's parentEnv chain traverses anon:I): the actual *exposure*. Sites 1 and 2 store the bound in two different envs but the duplicate only surfaces because of the parent-chain traversal.

Fix at site 3 — sever drop:0's parent chain so it doesn't pass through the citizen env.

### Why drop:0's parentEnv was anon:I

`StructCompiler.precompileStruct` builds `outerEnv = CitizenEnvironmentT(parent = declaringEnv, id = structTemplateId)` — `outerEnv` *is* the citizen's template env. `StructCompilerCore.compileStruct`'s first deferred-compile loop pairs lifted/sibling methods (drop, constructor, forwarders, lifted internal methods) with `structInnerEnv` as their compilation env, and `structInnerEnv.parentEnv = outerEnv`. So drop:0's chain went: drop:0 → structInnerEnv → outerEnv → packageEnv — through both citizen-shape envs.

The @SMLRZ comment "lifted methods go to outerEnv" was doing less than it suggests: `outerEnv` is itself a citizen env identified as the struct, so "lifted = outerEnv" still parented lifted methods to the citizen at the env-hierarchy level. What we actually want for UFCS is the env that *declared* the struct — `declaringEnv`, which is `outerEnv.parentEnv`.

### The change

Two-file change:

- `StructCompilerCore.compileStruct` takes a new `declaringEnv: IEnvironmentT` parameter — the env that declared the struct (already in scope at the call site as `structTemplata.declaringEnv`).
- The first deferred-compile loop pairs lifted/sibling methods with `declaringEnv` instead of `structInnerEnv`.
- `StructCompilerGenericArgsLayer.compileStruct` passes `declaringEnv` through.

```scala
// Before:
FunctionTemplataT(
  structInnerEnv, // Per @SMLRZ, compiled in inner env (instantiated struct ID)
  functionA))

// After:
FunctionTemplataT(
  // UFCS: lifted/sibling methods compile as free functions — their parent
  // chain skips the citizen env so lookup walk-up doesn't surface the
  // struct's own bound declarations as duplicate candidates.
  declaringEnv,
  functionA))
```

The second deferred-compile loop (which uses `if (functionA.lift) outerEnv else structInnerEnv`) is unchanged — non-lifted methods continue to compile against the inner env.

### Why this is principled

- Encodes UFCS philosophy: a method is a free function; `self`'s containing struct doesn't get privileged status in the env hierarchy.
- Removes the duplicate at its origin (env construction), not at consumer-side filtering.
- Doesn't touch lookup machinery, doesn't change the OverloadResolver assertion, doesn't change rich-name shape (the branch's @SMLRZ goal — `Vec<i32>.push()`-shaped paths — is preserved).
- Two envs in the chain (drop:0 → structInnerEnv → outerEnv) collapse to one (drop:0 → outerEnv.parentEnv = packageEnv); cleaner.

## Approaches considered (rejected, in order)

For posterity — when the obvious-looking fix doesn't work, knowing why saves the next person time.

### A — relax the assertion (template-strip both sides of `startsWith`)

Patch `OverloadResolver.scala:630` to template-strip each step before comparison. **Falsified**: a downstream check at `CompilerOutputs.scala:203` (the @MFBFDP "bound must be rooted inside the calling denizen" check) also failed, which proved the resolver was picking the wrong candidate. The assertion was structurally accurate; the relaxation hid it.

### B — pick the longest candidate instead of the shortest

drop:0's NBIFP-imported bound is the most-local declaration; downstream consumers (line 203) expect calling-denizen-rooted ids. **Partially worked** (175 → 180) but `foreach`/`stdlib_hashmap`/`stdlib_hashset` then hit a different `addInstantiationBounds` assertion (line 236, `vassert(existing == instantiationBoundArgs)`). Pick-longest just shifts which scenario gets the right answer; doesn't fix the underlying duplication.

### Pre-3 — fix env construction so `denizenTemplateId` is pure-template form

Template-strip parent steps in `evaluateGenericLightFunctionFromNonCall`'s id construction. **Partially worked** (175 → 180). But `Round 2` instrumentation revealed mixed-form ids are routine in the typing pass (53 unique mixed-form ids per minimum-repro compile, only 4 of them being the shape this fix targeted). And `CompilerOutputs`'s comparison sites already template-strip on both sides — the codebase convention is "tolerate mixed-form, normalize at compare time," not "produce pure template form everywhere." So this attempt was inconsistent with the existing convention and addressed only a narrow slice of the broader pattern.

### Approach C — filter bound-prototype candidates from `getParamEnvironments` source

Filter `FunctionBoundNameT` candidates out of param-env-sourced collection in `OverloadResolver.getCandidateBanners`, trusting NBIFP to import all needed bounds locally. **Falsified by data**: the `__call` duplication came entirely from the call-site env's own parent-walk (Source 1), not from param-env sources. Filtering Source 2 didn't fire on `__call` at all.

### Approach #1 — redirect `entryToTemplata` for lifted functions

When pairing a lifted `FunctionEnvEntry` with `definingEnv`, redirect to `definingEnv.parentEnv` to mirror StructCompilerCore's `lift=true` choice on the lookup-pairing path. **Didn't fire**: the two `__call` candidates aren't `FunctionEnvEntry` lookups — they're `TemplataEnvEntry(PrototypeTemplataT(FunctionBoundNameT))` lookups, which never go through `entryToTemplata`'s `case FunctionEnvEntry` arm. The path was the wrong layer.

### Lookup-walk filter (filter `PrototypeTemplataT(FunctionBoundNameT)` from parent-walk results when local has one)

Encodes the rule "local NBIFP-imported view shadows ancestral declarations." **Sound** under all three NBIFP completeness gaps surveyed (return-type bounds, transitive nested bounds, match-case-type-mention) — the filter only triggers when local has the bound, so gap cases preserve existing behavior. But user instinct rejected it: filters are downstream defense; the right fix is upstream prevention. That instinct led to investigating site 3, where the duplicate is *exposed*, not just where it shows up.

The applied fix (site 3 in `StructCompilerCore.scala`) is the upstream version: drop:0's parent chain doesn't traverse anon:I, so the substruct's contract bound never surfaces during walk-up. No filter; the data isn't reachable in the first place.
