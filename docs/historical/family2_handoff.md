## 1. The one-paragraph orientation

You're working in the **Vale compiler's postparser** (the pass that happens after the parser and before type-checking). You're fixing two tests about how lambdas handle untyped parameters. Some background worth reading first: per @LAGTNGZ (`docs/arcana/LambdasAreGenericTemplatesNotGenerics-LAGTNGZ.md`), Vale splits top-level functions (true generics: compile once, Instantiator stamps per-type later) from lambdas (templates: per-call-site expansion with argTypes baked into the function name). Both paths, however, depend on `FunctionS.genericParams` to be accurate — the typing pass reads it to build the per-call-site solve for lambdas identically to how it builds the one-shot solve for generics. The tests below fail because when the parser sees `(a, b) => body` or `(_) => body`, the postparser synthesizes an implicit rune for each untyped parameter but never lifts those runes into `genericParams`. Adding that step is probably the whole fix.

## 2. Prerequisites — what you need to know about Vale before starting

### 2.1 The passes

Source code flows through: **parser** → **postparser (a.k.a. "scout")** → **higher-typing** → **typing pass** → **instantiator** → **simplifier** → **backend**.

You care about **postparser** (primary fix target) and **instantiator** (worth sanity-checking, probably no changes needed there).

- Postparser input: parse-tree nodes (types ending in `P`, e.g. `LambdaPE`, `MagicParamLookupPE`, `ParameterP`).
- Postparser output: scouted AST (types ending in `S`, e.g. `FunctionS`, `ParameterS`, `IExpressionSE`).
- Typing-pass input: scouted AST + rune-type info from higher-typing.
- Typing-pass output: placeholdered typed AST (types ending in `T`).
- Instantiator output: monomorphized typed AST (types ending in `I`).

Every pass re-builds its own AST forms. Don't get confused when you see `FunctionA` (higher-typing), `FunctionS` (scout), `FunctionT` (typing), `FunctionI` (instantiator) — they're the same concept at different stages.

### 2.2 Runes, generic params, magic params

- **Rune**: a named type variable inside a function's `where` clause / implicit-type system. `func foo<T>(x T)` has rune `T`. Runes can be explicit (`T` is a `CodeRuneS("T")`) or implicit (generated by the compiler for anonymous types).
- **Generic parameter**: a rune that's **identifying** — that is, the compiler needs to know its value to pick which instantiation to call. `<T>` in `func foo<T>(x T)` is a generic param. In Vale's internal vocabulary, "identifying rune" and "generic param" are almost interchangeable — a generic param is a rune that also shows up in the function's external identity.
- **Magic param**: syntactic sugar for `_` in a lambda body. `(_ + 1)` means "a lambda that takes one argument and returns it + 1." The `_` is a reference to a parameter that was never explicitly declared. The postparser handles this by creating a synthetic `MagicParamNameS` for each unique magic param position and generating a fresh parameter for it.

Magic params predate generics. Under the old template system, `(_) => _ + 1` would expand at the call site and `_` would pick up the caller's type. Under generics, each magic-param position becomes its own generic parameter so the lambda compiles once with abstract placeholders.

### 2.3 LDNEIR — a design decision that's *relevant but not quite right*

The design doc at `docs/old/Parser_Scout.md:643` introduces **LDNEIR — Lambdas Don't Need Explicit Identifying Runes**. Paraphrased: *"Lambdas don't need to explicitly specify their identifying runes; we figure them out from the generics of the callee."*

This comment survives in the current FunctionScout code (`FunctionScout.scala:460-461`). It reads like "lambdas don't need generic params at all," but that's a misreading. The actual design is: **the user doesn't need to write `<X, Y>` on a lambda, but the compiler still needs to create identifying runes for untyped params internally.** The pre-refactor code did this; the post-refactor code stopped doing it. The comment is being used to justify an incorrect simplification.

Don't let the LDNEIR comment confuse you. It's about **user-facing syntax**, not about whether lambdas have generic params.

## 5. Suggested fix (Phase 1 — postparser)

The cleanest way is to mirror the pre-refactor structure: track which explicit params had their rune auto-synthesized vs. which were user-typed, and lift the former into generic params.

### 5.1 Plan

1. Update lines 248-289 to also track, alongside each `ParameterS`, whether its rune was synthesized (`None` → `Some(ImplicitRune)` branch) or user-supplied (`Some(_)` branch).
2. Emit a `GenericParameterS` for each synthesized-rune parameter.
3. Concat that list into `genericParametersS` at line 491-498.

## 8. Pitfalls and gotchas

### 8.1 `vimpl()` at end of tests is not your problem

Both test bodies end with `vimpl()`. That's intentional — the authors left room for additional assertions (see the commented-out lines in 2.2). If your fix makes `lambda.function.genericParams.size shouldEqual 1` pass but then `vimpl()` panics, that's **still success** from the Family 2 perspective. The test will still show as "failed" in the runner, because scalatest sees the panic. You may want to remove the `vimpl()` line *after* your fix makes the core assertions pass — but only with explicit approval, since removing it counts as changing the test.

### 8.2 The pre-refactor code referenced `identifyingRunes`, not `genericParametersS`

Don't confuse yourself by grepping for `identifyingRunes` in the current code — that name is gone. The current equivalent is `genericParametersS`. The concept maps 1:1.

### 8.3 `patternS.captureS` might be `None` for `_`, but that's not your distinguishing feature

A junior might be tempted to detect "`_` params" by checking `captureS.isEmpty` after translation. Don't go that route. **Detect on the rune side, not the capture side.** A param's rune gets synthesized iff the user didn't write an explicit type, regardless of whether the capture name is `_` or `a`. Both `(a, b) => ...` and `(_) => ...` have the same "untyped param" shape from the rune perspective. The sketch in §5.2 does this correctly.

### 8.4 Don't apply the fix to non-lambdas

`scoutFunction` is called for top-level funcs, interface methods, AND lambdas. Your lift-implicit-runes-to-generics step should be inside `maybeParent match { case ParentFunction(_) => ... }`. Top-level funcs with `func foo(a)` (untyped param) don't auto-generate generic params — that's a syntax error for non-lambdas. If you lift for all functions, you'll regress tests elsewhere in the codebase.

### 8.5 Regions filter

Lines 495-498 filter out `RegionGenericParameterTypeS` from the final generic-params list. Your new entries use `CoordGenericParameterTypeS`, which is not filtered. But double-check that detail matches what the rest of the codebase expects for lambda generic params.

### 8.6 The Rust port

There's a parallel Rust implementation at `FrontendRust/src/postparsing/function_scout.rs` with the same `LDNEIR` comment (line 698, 1219). Don't change the Rust code — the migration is intentionally parity-only and the Rust tracks Scala. A later migration pass will pick up your Scala change. The `.claude/CLAUDE.md` rule "Rust should mirror Scala as close as possible" applies.

### 8.7 Run-time bug surface is postparser only

The typing pass and instantiator are downstream of the postparser. Your fix changes the shape of `FunctionS` for lambdas (adds generic params). This flows into higher-typing and then the typing pass. The typing pass already handles generic functions; your extra generic params should "just work" through the existing machinery.

If something downstream breaks, the failure mode will be type-solver errors about unresolvable runes, or missing `runeToType` entries. Check that `runeToExplicitType += ((rune.rune, CoordTemplataType()))` is called for every rune you lift — it already is in the current code (line 280), so you should be fine, but verify.

## 9. Mental model for what you're doing

You're extending the "this lambda has a parameter with an unspecified type" mechanism from "generates a rune for internal type-checking" to "also generates a generic param so the lambda is actually generic over that type."

Right now a lambda like `(a) => a + 1` compiles to something internally like:
```
func lambda$0(closure &Closure$0, a:rune_42) ??? {
  a + 1
}
```
…where `rune_42` is an implicit rune that exists in the function's rule set but isn't in its *identifying* runes. So the typing pass sees this as a function with 0 generic params — and every call site tries to compile it with the same single signature. That works under templates (the body gets type-checked per call site), fails under generics.

After your fix, it compiles to:
```
func lambda$0<rune_42>(closure &Closure$0, a:rune_42) ??? {
  a + 1
}
```
…where `rune_42` is now a generic param. Each call site gives it a concrete value, and the Instantiator produces distinct `lambda$0<int>` and `lambda$0<bool>` versions naturally.
