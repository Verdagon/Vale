# PlaceholdersNamedByDenizenTemplate-PNBDTZ

When the TypingPass creates a placeholder (an unknown-until-solved type variable) for a generic parameter, it names the placeholder with a prefix derived from the *pre-lift template ID* of the enclosing denizen, not from the post-lift instance ID. The OWPFRD sanity check in `sanityCheckConclusion` must use `rootDenizenEnv.denizenTemplateId` (pre-lift) to validate the prefix, not `rootDenizenEnv.id` (post-lift).

For a user-defined `func drop(self Ok<OkType, ErrType>)` (see @SMLRZ for lift mechanics), `drop` is treated as a free function whose root denizen is `drop` itself — so `rootDenizenEnv.templateId.steps = [drop]`. Its placeholders are named with that prefix: `[drop]::KindPlaceholderName(0, OkType)`. After lift, the function's runtime ID becomes `[Ok<T,E>, drop(args)]` — but the placeholder's name prefix doesn't change. Comparing the placeholder `[drop]` against the lifted env `[Ok<T,E>, drop]` via `startsWith` would always fail (one is a prefix of the other only if we compare templates, not instances).

The check at `Frontend/TypingPass/src/dev/vale/typing/Compiler.scala` `sanityCheckConclusion` reads `rootDenizenEnv.denizenTemplateId` for this reason. Before the fix, it used `rootDenizenEnv.id` with a match that extracted the template name but kept `initSteps` intact — producing `[Ok<T,E>, drop(template)]`, which `[drop]` could never start with. This assertion fired on *every* program that went through the lift path, including `exported func main() {}` because builtin `Result`'s `drop` is lifted.

**How this affects the rest of the codebase:** Any future sanity check, substituter, or placeholder-identity comparison that walks from a value (a templata) back up to a root denizen environment must compare against `templateId`/`denizenTemplateId`, not `id`, for correctness across lifted vs. non-lifted functions. If you see `.id` used this way, check whether the code works when `drop(self Foo<T>)` is in play.
