# ClangArchDerivedFromRustTriple-CADFRTZ

The Coordinator passes `-arch arm64` (or `-arch x86_64`) to clang explicitly when Rust interop is active, derived from the Rust target triple Divination used. Without this, a Rosetta-translated parent process silently makes clang emit the wrong architecture, which then fails to link against Divination's Rust `.a`.

The failure mode is subtle: the bootstrap `valec` binary is x86_64 (it was built on/for x86_64). On Apple Silicon it runs under Rosetta. When it spawns clang, clang inherits some architecture preference from its parent — in our testing this manifested as `/usr/bin/clang` (a universal binary targeting both arches) dispatching to `ld` with `-arch x86_64`, even though `clang --version` from an arm64 shell reports `Target: arm64-apple-darwin23.4.0`. The result: clang compiles all the Vale C wrappers to x86_64 objects, then `ld` tries to link them with `libexample-project.a` (built arm64 because Divination's `cargo cbuild --target aarch64-apple-darwin` hardcoded the right triple) and either hangs or errors.

In `Coordinator/src/clang.vale::invoke_clang`, the triple is recovered from the `rust_include_dir` path — Divination writes artifacts to `<rust_output_dir>/target/<triple>/release/`, so the parent directory name of `rust_include_dir` IS the triple. We don't pass the triple as its own argument (an `Opt<str>` reassignment pattern triggers a bootstrap compiler bug — see `docs/migration/rust-interop.md`), we just derive it inline. The triple is mapped to `-arch` only for Darwin: `aarch64-apple-darwin` → `arm64`, `x86_64-apple-darwin` → `x86_64`. Other triples (Linux, Windows) don't use `-arch` the same way and get no override.

**How this affects the rest of the codebase:** when the bootstrap `valec` eventually becomes arm64-native (or whenever we rebuild it as a universal binary), the Rosetta-induced arch inheritance goes away and this explicit `-arch` becomes a no-op on single-arch machines — but still correct. It's not safe to *remove* the flag because any future pipeline that runs through Rosetta (cross-compilation rigs, CI matrixes) would regress silently. If you change the clang invocation site, preserve this behavior. Also: the triple-detection in Divination (from `rustc -vV`) and the triple-recovery in Coordinator (from directory name) are two sides of the same contract — if you change the output-directory layout Divination uses, update the Coordinator recovery.
