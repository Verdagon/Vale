# DivinationStdoutToLogFileNotPipe-DSLFNPZ

The Backend's `doRustyThings()` invokes Divination via `std::system(cmd)` with shell redirection `> <rust_output_dir>/divination.log 2>&1` appended. Divination's stdout/stderr is captured to a file rather than inherited from the parent, deliberately — if it inherits and any upstream parent process (e.g. the Coordinator) tries to capture the Backend's stdout via a pipe, Divination's `cargo cbuild` output overwhelms the Vale stdlib's subprocess reader and deadlocks everything.

The Vale stdlib's `stdlib::command::Subprocess` reads child stdout via `read_into_buffer` in `subprocess.c`, which calls `fgetc(stream)` one character at a time in a loop of up to 1000 chars per read. `consume_and_join` drains stdout and stderr alternately. When `cargo cbuild` rebuilds core/std/alloc under `-Zbuild-std`, it produces megabytes of output — the char-at-a-time reader can't keep up with the producer, pipe buffers fill, the writer blocks on PIPE_BUF backpressure, and the parent's read-then-check-alive loop never progresses because alive stays true while a subprocess is waiting to write. The Coordinator would hang forever after "Compiling panic_unwind v0" with no further progress.

Redirecting to a file sidesteps the problem entirely: Divination writes to a filesystem buffer (OS-managed, megabytes capacity), Backend sees nothing on its own stdout from Divination, and the Coordinator's read of Backend stdout stays small and drainable. The log file stays on disk for debugging.

**How this affects the rest of the codebase:** any future subprocess call inside the Backend (or any deep-pipeline callee whose output is captured by a parent via a Vale stdlib Subprocess) that produces more than roughly 16 KiB of output before exiting should either redirect to a file like we do here, or the Vale stdlib's `read_into_buffer` needs to be fixed to use block reads. Until the stdlib fix lands, prefer the file redirect pattern. See `docs/reasoning/rust-interop.md` for why we chose file redirect over fixing the stdlib.
